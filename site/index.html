Why embed it

Under your brand

White-label by design. Your UI, your flow, your support workflow — Casual RAS is the engine underneath.

Consent-based sessions

Built to embed

How consent works

The local user is always in control

  1. Held until Allow. A connecting viewer is paused in the handshake — no screen data is sent — until the local user approves.
  2. Approve or refuse. The prompt shows the peer’s identity. Deny, or 90 seconds of silence, refuses the session fail-closed.
  3. Always visible. While viewing is active a “remote viewing active” indicator stays on screen — the interface can’t hide it.
  4. Stop anytime. The stop control is always present and ends the session immediately.

Security model

Secure by default, responsive under load, honest in the interface

Our engineering priorities, in order: security first, responsiveness second, interface richness third. When they conflict, the higher one wins.

Authorization at the host

Encryption secures the pipe; permission is decided by the host, not the transport. A viewer requests — it never self-authorizes.

Responsive under load

A stalled video never freezes the viewer’s own pointer or the stop control. Latency beats interface richness by design.

Honest by construction

Active viewing is always visible and the stop control is always present. The product build cannot even link a “skip consent” path.

Technical details

Invariants, dependencies & FAQ

Non-negotiable security invariants
  • The local user is the final owner. A viewer requests; it never self-authorizes.
  • Emergency stop overrides everything — grant, lease, in-flight state — within the target time.
  • Consent is honest and unspoofable. Active viewing is always visible; branding may not hide it.
  • Transport encryption is necessary, not sufficient. Authorization is enforced by the host.
  • Secrets never touch logs — keys, tokens, clipboard, typed text or screen pixels.
  • Capability scope is enforced host-side, per message — never trust a client’s claimed scope (roadmap).
Dependency & license policy
Apache-2.0 across the whole repository, including a patent grant, so it is safe to embed in proprietary applications. Copyleft licenses (GPL/LGPL/AGPL/SSPL) are rejected and blocked at build time by an automated license gate; permissive licenses (MIT/Apache/BSD/ISC/Zlib/MPL-2.0) are allowed.
Does it need a server?
No application backend is required for the MVP. Direct peer-to-peer connections are preferred; when a direct path is unavailable, an encrypted relay carries the traffic. There is no central application server in the middle.
Can a viewer control my keyboard and mouse?
Not in the alpha. It is view-only: the viewer sees the shared screen and points at it with a visual cursor. No clicks or keystrokes are injected into the shared machine’s OS.
Which platforms can share vs. view?
Viewing works on macOS, Linux and Windows. Sharing is macOS-first in the alpha; Linux (PipeWire) and Windows (Desktop Duplication) sharing backends are next.
Is there an SDK yet?
Not yet. The approach is app-first: prove the roles in a real reference app, then draw the SDK boundary (C ABI + Node.js) around the proven crates. An SDK surface can’t be validated without a real consumer.
