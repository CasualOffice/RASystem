What it is

A remote-access engine you embed, not a product you send users to

The end products are a native host runtime (embedded on the controlled machine), a controller app for the support side, and a shared Rust core that later becomes the SDK.

🧩

Embeddable & white-label

Drop it into your application. Your brand, your flow — not a redirect to a third-party remote-desktop app.

🔗

Peer-to-peer over Iroh/QUIC

Encrypted, NAT-traversing, relay-fallback transport. The two machines connect directly — no central server in the MVP.

🦀

Rust core + Tauri

One shared Rust core drives both ends; video renders through WebCodecs in a Tauri webview. The SDK is drawn around the proven crates later.

✅

Consent-first by design

The local user authorizes every session. No pixels leave the machine until they click Allow — and the stop control is always present.

How it works

One app, both roles

The same download can share this screen or connect to another. Pick a role on the home screen.

Share a ticket

The person sharing picks “Share this screen” and gets a connection ticket (CASUALRAS1:…) to send over any channel.

Connect & approve

The viewer pastes the ticket and connects. The sharer sees an Allow / Deny prompt with the peer’s identity — nothing streams until they approve.

View & point

The screen renders on the viewer. Their cursor shows on the sharer’s screen as a “look here” pointer — collaboration without remote control.

🖥️ Share (agent)

Captures this machine and streams it, behind local consent, with an always-visible “remote viewing active” indicator and a stop control. macOS today; Linux & Windows on the roadmap.

👁️ Connect (viewer)

Decodes the H.264 stream with WebCodecs to a canvas, adds annotation tools and a remote pointer. Decode-only, so it runs on macOS, Linux, and Windows today.

Security

Security beats latency. Latency beats UX.

That ordering is a decision rule, not a slogan. When two conflict, the higher one wins — every time.

1
Security — never skip consent, grant validation, or an audit write to save milliseconds.
2
Latency — a stalled video must never freeze the controller’s own pointer or the stop button.
3
UX — richer but slower loses to a responsive, honest interface.

Non-negotiable invariants (a sample)

  • The local user is the final owner. A controller requests; it never self-authorizes.
  • Emergency stop overrides everything — grant, lease, in-flight input — within the target time.
  • Consent is honest and unspoofable. Active viewing is always visible; white-labeling may not hide it.
  • Transport encryption is necessary, not sufficient. Authorization is enforced by the host, not the pipe.
  • Secrets never touch logs — keys, tokens, clipboard, typed text, or screen pixels.
  • No copyleft in the graph. GPL/LGPL/AGPL/SSPL are blocked at build time.

We’re honest about limits. This is an alpha. Authorization is still coarse (real local Allow/Deny, but no signed grants or capability scoping yet), sharing is macOS-only, and builds are unsigned. The planned on-device fraud-friction subsystem is designed to deter coached-victim scams and contain remote attackers — it will never claim to “prevent scams” or be tamper-resistant.

Questions

Frequently asked

Is it really peer-to-peer?
Yes. The two machines connect directly over Iroh/QUIC — an encrypted, NAT-traversing transport with relay fallback. A connection ticket carries the host’s identity and addresses so the viewer dials it directly; there is no central server in the MVP.
Can a viewer control my keyboard and mouse?
Not in the alpha. It is view-only: the viewer sees the shared screen and points at it with a visual “look here” cursor. No clicks or keystrokes are ever injected into the shared machine’s OS.
How is consent enforced?
The local user is the final owner. A connecting viewer is held in the handshake — no pixels are sent — until the local user clicks Allow; Deny or 90 seconds of silence refuses fail-closed. The product build does not even link a “skip consent” path.
Which platforms can share vs. connect?
Connecting (viewing another machine) works on macOS, Linux, and Windows because the viewer only decodes video. Sharing your own screen is macOS-only in the alpha; Linux (PipeWire/VAAPI) and Windows (DXGI/Media Foundation) capture backends are on the roadmap.
What license is it under, and can I embed it commercially?
Apache-2.0 across the whole repository, including a patent grant — so you can embed it in proprietary apps. GPL/LGPL/AGPL/SSPL dependencies are rejected and blocked at build time.
Is there an SDK?
Not yet. The strategy is app-first: build two working reference roles that share Rust crates directly, then draw the SDK boundary (C ABI + N-API) around the proven crates. You can’t validate an SDK surface without a real consumer.
